## 1. Codebase Study

### What Hermes Is

A self-improving AI agent framework by Nous Research. Runs persistently on local hardware or cloud (Modal, Daytona, Docker). Reaches users via 17 messaging platforms. Works with any OpenAI-compatible LLM. Learns over time via skills and memory.

### Architecture Summary

Five layers:

| Layer | Key Files |
|-------|-----------|
| Agent core | `run_agent.py` (AIAgent), `agent/prompt_builder.py` |
| Tools | `tools/` — 44 modules, self-registering at import |
| CLI/TUI | `cli.py`, `hermes_cli/` (38 modules) |
| Gateway | `gateway/run.py` + 17 platform adapters |
| Persistence | `hermes_state.py` (SQLite FTS5) |

### Key Design Decisions

- **Ephemeral prompts** — system prompt rebuilt every turn; enables Anthropic prompt caching
- **Tool self-registration** — `registry.register()` at import time, no central list to maintain
- **OpenAI SDK for all providers** — swap LLM with config, no code change
- **Context compression at 85%** — fast auxiliary model (Gemini Flash) summarizes middle turns
- **Skills = procedural memory** — `SKILL.md` injected into prompt; full patch history + rollback (in this fork)

### Scale

- ~12.6K Python lines (core)
- 520 test files across 18 test directories
- 28 bundled skill categories + 14 optional
- 6 terminal backends (local/Docker/SSH/Modal/Daytona/Singularity)

## 2. How Hermes Self-Improves (Architecture)

Three mechanisms:

### 2a. Skills — Procedural Memory

Skills are Markdown files (`SKILL.md`) injected into the system prompt every turn. The agent can write, patch, edit, and delete its own skills using the `skill_manage` tool.

**Lifecycle:**
```
Agent solves hard problem (5+ tool calls)
  → calls skill_manage(action="create", ...)
  → SKILL.md written to ~/.hermes/skills/<name>/
  → prompt cache invalidated
  → next turn: skill injected into system prompt
  → agent "knows" that procedure permanently
```

Every `edit` or `patch` appends to `SKILL_HISTORY.md` (append-only). Rollback = new history entry. Agent-created skills pass a security scan (`skills_guard.py`) before activation.

### 2b. Memory — Declarative Memory

Two files injected into every system prompt:

| File | Contents | Char limit |
|------|----------|-----------|
| `MEMORY.md` | Facts about environment, projects, tool quirks | 2,200 |
| `USER.md` | User preferences, communication style, habits | 1,375 |

Priority tiers: `[HIGH]` (never evicted), `normal`, `[EPHEMERAL expires=YYYY-MM-DD]` (auto-drops after date).

Snapshot frozen at session start (preserves Anthropic prefix cache). New writes appear in "Added this session" block immediately; permanent in next session.

External backends (Honcho, Mem0, etc.) via `MemoryManager`. `prefetch_all()` recalls before each turn. `sync_all()` persists after.

### 2c. Context Compression — Working Memory Extension

At 85% context limit, `ContextCompressor` triggers: prune old tool outputs → protect head + recent tail → auxiliary model summarizes middle turns (structured: Goal / Progress / Decisions / Files / Next Steps) → summary replaces middle turns in-memory, originals saved to SQLite.

## 3. Does the Self-Improvement Actually Work?

### The Core Finding

All five self-improvement claims reduce to the same pattern:

> A prompt string tells the LLM to call a tool. The tool writes to disk. Disk state reloads in future sessions.

No background process. No outcome measurement. No gradient updates. Whether improvement happens depends entirely on **whether the LLM decides to call the tool on that turn**.

This is better described as **LLM-driven self-annotation** than a built-in learning loop.

### Claim-by-Claim Reality Check

| Claim | What Actually Happens | Verdict |
|-------|----------------------|---------|
| Creates skills from experience | `SKILLS_GUIDANCE` string → LLM *may* call `skill_manage` after 5+ tool calls | Partial |
| Improves skills during use | Same prompt instructs LLM to patch if it notices errors | Partial |
| Nudges itself to persist knowledge | `MEMORY_GUIDANCE` string present every turn — that is the nudge | Weak |
| Searches past conversations | FTS5 keyword search + Gemini Flash summarization | Strong |
| Builds deepening user model | 1,375-char freeform text file. No inference, hard ceiling | Weak |

### Key Gaps

**No success signal.** Skills created based on tool call count (≥5), not task outcome. A skill can be created from a failed or partially correct workflow.

**No patch history** *(fixed in this fork — upstream has nothing).* Original code: patch = destructive overwrite. This fork added `SKILL_HISTORY.md` + rollback CLI.

**Mid-session blindness.** Memory snapshot frozen at session start. User reveals important context at turn 5, asks related question at turn 15 — Hermes won't see the new memory until next session.

**Semantic search missing.** FTS5 is keyword-only. "slow API calls" won't find a session that diagnosed "high latency HTTP requests." No embeddings, no semantic re-ranking.

**LLM volition is the only trigger.** If the model doesn't create a skill on a complex turn, no skill is created. No background scheduler reviews sessions.

**USER.md ceiling.** 1,375 chars ≈ 10–15 facts. "Deepening model" stops deepening.

**No quality gate.** Skills created from bad workflows persist forever unless manually deleted.

**No deduplication.** Agent may create two similar skills with different names across sessions.

### What Actually Works Well

- **Session search** (FTS5 + Porter stemming + parallel Gemini summarization) — genuinely useful, solid implementation
- **Skill file pipeline** (atomic write → security scan → cache invalidation) — robust mechanics
- **Context compression at 85%** — enables longer work sessions without losing thread
- **Memory persistence** — survives crashes, written immediately to disk
- **Honcho integration** (opt-in) — actual dialectic modeling, closest to real user modeling; disabled by default

### The Gap That Would Close It

A **background scheduled process** — not dependent on the LLM's in-context volition — that reviews sessions, measures outcomes, and applies improvements automatically. The existing architecture (SQLite sessions, SKILL_HISTORY.md, skill_manage tool) is already the right substrate. What's missing is the loop that runs on top of it.

## 4. Karpathy Autoresearch Loop: Design Discussion

### What the Loop Requires

Karpathy's autoresearch loop (as applied to an agent system):

1. **Hypothesis representation** → skills (`SKILL.md` files)
2. **Experiment runner** → conversations (sessions in `state.db`)
3. **Automatic evaluator** → the hard part; no humans per iteration
4. **Hypothesis updater** → `skill_manage(action="patch")`
5. **Fast iteration** → nightly, not monthly

The bottleneck is always evaluation speed. Human evaluation = human-speed learning. Automated evaluation = 100x faster.

### Design Decisions Agreed In Session

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Autonomy | Fully autonomous | Fastest loop; regression watch handles rollback |
| Reward signal | Token efficiency + absence of correction + task completion | Combined signal; token count alone is gameable |
| Karpathy angle | Both self-play eval + real session signal | Self-play gates changes; real sessions confirm them |
| LLM in cron | Yes, use freely | Quality > cost at nightly frequency |

### The Three-Part Reward Signal

| Signal | Measurement | Source |
|--------|-------------|--------|
| Token efficiency | `(session_tokens - baseline) / baseline` | Session metadata |
| Absence of correction | Regex: "no", "wrong", "try again", goal rephrase | Session transcript |
| Task completion | Session ended naturally + acknowledgment final message | Session transcript |

### Five Design Directions Explored

**A — The Nightly Scientist** *(primary direction)*  
Treat each skill as a falsifiable hypothesis. Real session signals identify underperforming skills → LLM generates candidate patches → self-play evaluation gates application → auto-apply winners → regression watch rolls back losers.

**B — The Adversarial Red-Teamer**  
Actively try to break each skill: generate adversarial/edge-case scenarios, run agent against them, measure failure rate. Red-teaming surfaces gaps that normal sessions never encounter.

**C — The Pattern Miner (Gap Detection)**  
Cluster sessions by task type. Find high-frequency clusters with no matching skill. Auto-generate new skills from best session in each uncovered cluster. Proactive complement to Direction A.

**D — The Counterfactual Replayer**  
Take failed sessions. Replay with candidate improved skill. If replay succeeds where original failed, apply patch. Most rigorous evaluation; most expensive — reserve for high-value skills.

**E — The ELO Tournament (Long-horizon)**  
Build a benchmark from real sessions. Each night, run all candidate skill versions against benchmark. ELO-rate each version. Winning version becomes canonical. Needs 30+ days of history.

**Recommended combination:** A (core loop) + C (gap detection) + B (adversarial testing). D as gold-standard gate for high-invocation skills. E as long-horizon overlay.

### Phase 4: Self-Play Evaluation (The Karpathy Gate)

For each candidate patch:

1. Generate 5 synthetic task scenarios by mutating real session task descriptions (grounded in actual distribution, not random)
2. Run agent twice per scenario: old skill vs. new skill (using cheap model — Gemini Flash or Haiku)
3. Evaluate: `token_delta = (new_tokens - old_tokens) / old_tokens` + LLM judge quality score (two independent judges, require agreement within 2 points)
4. Accept if: `token_delta < 0` AND `quality_delta ≥ 0`. Reject if quality drops. Hold for human review if judges disagree.

### Key Assumptions to Validate

- Token count is a valid proxy for efficiency (only valid when correction_rate is simultaneously low)
- Self-play synthetic tasks represent real task distribution (mitigate: mutate real session descriptions, not random generation)
- LLM-as-judge is calibrated (mitigate: two independent judges + objective token metric tiebreaker)
- Auto-patching can't degrade skills (mitigate: SKILL_HISTORY.md rollback + Phase 6 regression watch)
- Correction signal is detectable from transcript (mitigate: detect implicit goal-rephrase pattern, not only explicit "wrong")

### MVP Scope

1. Phase 1 — session signal extraction to `skill_metrics.db`
2. Phase 2 — underperforming skill detection only
3. Phase 3 — LLM patch generation for worst-performing skill
4. Phase 4 — 3 synthetic tasks, 1 judge, simple token+quality gate
5. Phase 5 — auto-apply with `SKILL_HISTORY.md` logging
6. Phase 6 — next-night rollback if `correction_rate` rises >15%

Out of MVP: missing skill detection, memory staleness repair, ELO tournament, multi-day trend analysis, semantic deduplication.

## 5. Coexistence of Two Improvement Systems

### The Problem Discovered

After designing the autoresearch loop, a critical conflict was identified: both in-session LLM patching and the autoresearch cron job write to the same `SKILL.md` files with no coordination. Without safeguards they silently overwrite each other.

Three concrete failure modes:

**Sequential overwrite:** Autoresearch runs at 3am using yesterday's session data. The LLM patched the same skill at 2pm yesterday, fixing the root cause. Autoresearch — working from pre-fix signal — generates and applies a conflicting patch. The LLM's fix is silently lost.

**Regression watch misattribution:** Autoresearch patches skill X at 3am. LLM patches the same skill at 9am (makes it worse). Regression watch runs next night, sees correction_rate rose after the 3am patch, rolls back the 3am patch. The 9am LLM patch (the actual cause) survives untouched.

**Stale signal contamination:** Monday morning a skill performs badly. LLM fixes it Monday afternoon. Tuesday's autoresearch extracts Monday's aggregated signals — which include the bad pre-fix sessions — flags the skill as underperforming, generates a patch for something already fixed.

### Existing History Mechanism vs. Proposed Loop

The fork already has `SKILL_HISTORY.md` + `hermes skills history` + `hermes skills rollback`. This is **not** a learning loop — it is a human-readable audit trail. Nothing in the codebase reads `SKILL_HISTORY.md` programmatically to drive improvements. The proposed autoresearch loop is complementary, not redundant.

| | Existing history mechanism | Proposed autoresearch loop |
|--|---------------------------|---------------------------|
| Trigger | LLM patches in-session (prompted by `SKILLS_GUIDANCE`) | Nightly cron, independent of LLM volition |
| Signal | None — LLM acts on in-context observation | Measured session outcomes (token count, correction rate, completion) |
| History role | Write-only audit trail, read by human CLI | Safety net for regression rollback |
| Improvement driver | LLM's in-session judgment | Measured underperformance |

### Resolution: Hotfix vs. Release Model

**Mental model:** in-session patches = hotfixes. Autoresearch patches = releases. Releases build on hotfixes, never over them.

**Four coordination mechanisms:**

1. **Recency lock** — autoresearch skips any skill patched in-session within the last 24h. Prevents racing with fresh hotfixes before they accumulate signal.

2. **In-session patch rate as signal** — if a skill has been patched ≥3 times in-session in 7 days, autoresearch interprets this as structural gaps and proposes a comprehensive rewrite (incorporating all hotfixes) rather than another incremental patch.

3. **Source tagging** — every `SKILL_HISTORY.md` entry is tagged `[in-session]` or `[autoresearch]`. Requires adding one optional `source` parameter to `_append_skill_history()` in `tools/skill_manager_tool.py` — the only existing file that needs changing.

4. **Scoped regression watch** — Phase 6 only rolls back patches it made. If in-session patches occurred between the autoresearch patch and the measurement, it flags for human review instead of auto-rolling-back (ambiguous causation).

### The Resulting Relationship

The two systems become complementary:
- High in-session patch rate on skill X → autoresearch detects structural gap → proposes rewrite
- Good autoresearch rewrite → LLM stops needing to hotfix → in-session patch rate drops
- Dropping patch rate → autoresearch confirms the rewrite succeeded

This is measurable closed-loop validation.

## 6. Open Questions

1. **task_type classification cost** — embed at session-end (cheap, amortized) or at cron time (bursty)?
2. **skill_metrics storage** — new table in `state.db` or separate `skill_metrics.db`?
3. **low-invocation skills** — <3 uses in 30 days has no statistical signal; skip or self-play-only?
4. **digest delivery** — primary gateway channel or separate admin channel?
5. **synthetic task mutation strategy** — paraphrase only? Add noise? Introduce edge cases deliberately?
