# Session persistence

Every conversation Hermes has is saved to a local SQLite database with full-text search. Sessions are searchable, replayable, and used for RL training data generation.

## What it is

`SessionDB` (in `hermes_state.py`) is a SQLite database at `~/.hermes/state.db` that stores all conversation sessions. It uses the FTS5 (Full-Text Search 5) extension to make every word of every conversation searchable in milliseconds.

## The session record

Each session record stores:

| Field | Description |
|-------|-------------|
| `id` | UUID, generated at session start |
| `title` | Auto-generated summary (e.g., "Debugging nginx 502 error") |
| `created_at` | Timestamp |
| `updated_at` | Last message timestamp |
| `messages` | Full conversation history as JSON |
| `compressed` | Whether the session has been context-compressed |
| `platform` | Where the conversation happened (cli, telegram, discord, etc.) |
| `profile` | Which Hermes profile was active |

## Full-text search

The FTS5 index covers every message's content. You can search past conversations from inside a session or via the CLI:

```bash
hermes search "nginx config"
hermes search "the API key for"
hermes search "how to deploy"
```

From inside a conversation:

```
/search nginx 502
```

The search returns snippets from matching sessions, with context. Hermes can then summarize or reference the found material.

## Trajectory logs

In addition to the session database, every session is logged to a JSON file:

```
~/.hermes/logs/session_YYYYMMDD_HHMMSS_UUID.json
```

These trajectory logs capture the full conversation in the OpenAI message format, including tool calls and results. They're used for:

- **Debugging** — inspect exactly what the agent did and why
- **Replay** — re-run a conversation to reproduce an issue
- **RL training** — batch trajectory generation for fine-tuning tool-calling models

## Session titles

Session titles are auto-generated by `agent/title_generator.py` using a short LLM call after the first exchange. The title describes what the session was about, making it easy to find in search results.

You can override the title:

```
/title "Setting up the CI pipeline for frontend"
```

## Browsing sessions

From the CLI:

```bash
hermes sessions           # List recent sessions
hermes sessions search "query"   # Search sessions
hermes sessions show <id>  # Show a specific session
```

From inside a conversation:

```
/sessions          # List recent sessions
/search <query>    # Search sessions
```

## Context compression and sessions

When the conversation context is compressed (see [agent loop](agent-loop.md)), the original full conversation is preserved in the session record. Only the in-memory message list is shortened. You can always recover the full history from the database.

## Using sessions for RL training

`batch_runner.py` can generate large batches of trajectories from session data, compressing them into training format for fine-tuning language models. This connects to the Atropos RL training framework via `rl_cli.py`.

```bash
hermes rl generate-trajectories --sessions recent --output ./training-data/
```

## Database location

The default location is `~/.hermes/state.db`. It follows `HERMES_HOME`:

```bash
HERMES_HOME=/custom/path hermes    # Uses /custom/path/state.db
```

## Backup

To back up all sessions:

```bash
cp ~/.hermes/state.db ~/backups/hermes-$(date +%Y%m%d).db
```

SQLite databases are safe to copy while Hermes is not running. For live backups while Hermes is running, use SQLite's backup API:

```bash
sqlite3 ~/.hermes/state.db ".backup ~/backups/hermes-backup.db"
```
